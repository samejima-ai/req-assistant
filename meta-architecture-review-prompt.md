# プラットフォーム設計俯瞰レビュー用メタプロンプト

## コンテキスト
システム設計をメタ的に行うプラットフォームアプリの設計に対する包括的なレビューを実行する。メタモデルの整合性と、システムを生み出すシステムとしての強牢性を評価する。

## 評価観点

### 1. メタ・アーキテクチャの健全性
- プラットフォーム自体のアーキテクチャと、生成される対象システムのアーキテクチャが明確に分離され、レイヤー間の意図しない干渉が発生していないか。
- メタモデル（コンポーネント、関係性、制約）の抽象度が適切か。過剰な抽象化によってパフォーマンス低下や理解難易度の増大を招いていないか。

### 2. ワークフローとオーケストレーションの評価
- ユーザーがシステムをモデリング・設計する際のエージェンティック・ワークフローは直感的であり、かつ論理的な破綻がないか。
- 複数要素（UIレイヤー、ビジネスロジック、データ層）を連携し統合管理するオーケストレーションの仕組みが一貫性を保っているか。

### 3. ベストプラクティスとコンテキスト駆動開発の適合
- プラットフォームが生成・定義するモデルが、現代的なベストプラクティスに沿った設計を自然に誘導できる構造になっているか。
- 設計対象のビジネス要件やコンテキストの変化に柔軟に追従できるコンテキスト駆動開発（Context-Driven-Development）の思想が、プラットフォームのデータ構造自体に組み込まれているか。

### 4. ルールとスキルの拡張性
- 動的なコード生成や設計バリデーションを実行する際のエージェンティック・ルールが、ハードコードされずにユーザー側または設定ファイルから柔軟に定義・変更可能か。
- 外部サービス連携、AIによる自動補完、サードパーティ製プラグインの導入など、将来的なエージェンティック・スキルの拡張を見据えたインターフェース（APIやWebhook）が担保されているか。

## 出力指示
- 発見された設計上の矛盾（例: 自己言及的な依存関係、責務の重複）やボトルネックを列挙すること。
- 改善案を提示する際は、具体的なデータ構造、インターフェース定義、またはアーキテクチャ図をコードブロックを用いて明示すること。
